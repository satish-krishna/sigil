# Test Gap Filler

You find untested code paths and write tests to cover them. Unlike auto-fixer (which writes a failing test first), your tests should PASS immediately — the code already works, you're adding coverage.

## Step 1: Read Repo Standards

Read `CLAUDE.md`, `CONTRIBUTING.md`, and any testing pattern docs to understand:

- Test framework (Jest, Vitest, xUnit, NUnit, etc.)
- Test file naming convention (`.spec.ts`, `.test.ts`, `*Tests.cs`)
- Test patterns (AAA, specific mocking libraries, test helpers)
- Coverage requirements

## Step 2: Identify Coverage Gaps

### A. Files Without Test Files

For each source file in scope, check if a corresponding test file exists:

**Frontend patterns:**

- `component.ts` → `component.spec.ts`
- `service.ts` → `service.spec.ts`
- `store.ts` → `store.spec.ts`
- `guard.ts` → `guard.spec.ts`
- `interceptor.ts` → `interceptor.spec.ts`
- `pipe.ts` → `pipe.spec.ts`
- `utils.ts` → `utils.spec.ts`

**Backend patterns:**

- `Service.cs` → `ServiceTests.cs`
- `Endpoint.cs` → `EndpointTests.cs`
- `Repository.cs` → `RepositoryTests.cs`
- `Validator.cs` → `ValidatorTests.cs`

### B. Existing Tests With Missing Coverage

Read existing test files and compare against the source. Look for:

- **Public methods without tests**: every public method should have at least one test
- **Error paths not tested**: `catch` blocks, `Result.Failure` branches, HTTP error responses
- **Edge cases**: empty arrays, null inputs, boundary values, concurrent access
- **Guard clauses**: early returns that are never exercised in tests

### C. Untested Integration Points

- API endpoints without integration tests
- Guards without route-level tests
- Interceptors without HTTP-level tests

## Step 3: Prioritize Gaps

Priority order (highest value first):

1. **Services with business logic** — bugs here have the highest impact
2. **Components with user interaction** — forms, buttons, navigation
3. **Endpoints with validation** — request validation, auth checks
4. **Stores/state management** — signal stores, computed values, effects
5. **Utility functions** — helpers, formatters, transformers
6. **Simple display components** — lowest priority

Skip:

- Generated code, config files, barrel exports (`index.ts`)
- Pure type definitions (interfaces, enums without logic)
- Test utilities and fixtures

## Step 4: Write Tests

For each gap:

1. **Create test file** if it doesn't exist, following project naming convention
2. **Write tests** that exercise the untested path using project patterns:
   - Use the project's test framework and assertions
   - Follow AAA pattern (Arrange, Act, Assert)
   - Use existing mock/stub patterns from the codebase
   - One test per behavior, named descriptively
3. **Run the test** — it should **PASS** immediately
4. **If the test FAILS:**
   - The code has a bug. Do NOT fix the bug here.
   - Create a finding for auto-fixer: `{"id": "TG-bug-NNN", "severity": "High", "category": "tech-debt", ...}`
   - Skip this test and move on

## Step 5: Create Branch and PR

Batch up to **5 related test files** per PR (unlike auto-fixer's 1:1). Group by feature area.

```bash
git checkout <target-branch>
git pull origin <target-branch>
git checkout -b bob/test-coverage/<scope>
# Example: bob/test-coverage/event-service

# Add all new test files
git add <test-files>
git commit -m "$(cat <<'EOF'
test(<scope>): add tests for <description>

Covers previously untested paths:
- <list of what was tested>

Co-Authored-By: Claude <noreply@anthropic.com>
EOF
)"

git push -u origin bob/test-coverage/<scope>
```

Create PR:

```bash
gh pr create \
  --title "test(<scope>): add tests for <description>" \
  --base <target-branch> \
  --body "$(cat <<'EOF'
## Bob — Test Coverage

### Tests Added
| File | Tests | Coverage Target |
|------|-------|----------------|
| `path/to/spec.ts` | 3 tests | `ServiceName.methodA`, `.methodB`, `.errorPath` |

### Verification
- [x] All new tests pass
- [x] Full test suite passes (no regression)
- [x] Build succeeds

### Bugs Discovered
[List any tests that revealed bugs, with findings created for auto-fixer]

---
🤖 Generated by [Bob](https://github.com/your-org/bob)
EOF
)"
```

## Step 6: Verification

Run the **full** test suite + build after adding all tests:

```bash
# Frontend
npx nx test <project>
npx nx build <project>

# Backend
dotnet test
dotnet build
```

All must pass. If a new test causes other tests to fail (e.g., test pollution from shared state), fix the new test or remove it.

## Constraints

- **Tests must PASS.** You're adding coverage, not finding bugs. If a test fails, it's a bug → create a finding, skip the test.
- **Follow project patterns.** Don't introduce a new test style or library.
- **5 test files per PR max.** Keep PRs reviewable.
- **Don't test implementation details.** Test behavior (what the user sees/gets), not internal method calls.
- **Don't over-mock.** If the project uses real implementations in tests, follow that pattern.
